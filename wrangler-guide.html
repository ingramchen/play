⚡ Cache API vs KV — What's the difference?
Cache API (built-in)
  • • No binding needed — caches.default
  • • Stores HTTP Response objects
  • • Expires via Cache-Control header
  • • May be evicted anytime
  • • Per-colo (not globally consistent)
const cache = caches.default;
let res = await cache.match(req);
if (!res) {
  res = await fetch(origin);
  ctx.waitUntil(cache.put(req, res.clone()));
}
KV (binding required)
  • • Declare in wrangler.toml → env.APP_KV
  • • Stores string / ArrayBuffer / stream
  • • Optional TTL via expirationTtl
  • • Persistent — never auto-evicted
  • • Globally consistent (eventual)
const val = await env.APP_KV.get("key");
await env.APP_KV.put("key", "val");
// with TTL:
await env.APP_KV.put("k", "v", {
  expirationTtl: 3600
});
⚠️ KV Gotchas
  • • Eventually consistent — 寫入後最多 60 秒才全球生效
  • • 適合讀多寫少，不適合頻繁寫入
  • • 沒有 filter/query，只能 list({ prefix })
  • • 單個 value 上限 25MB
  • • Free: 1,000 writes/day, 100,000 reads/day
✅ 適合：設定檔、feature flags、靜態資產 metadata
❌ 不適合：session store、計數器、即時狀態
⚠️ D1 Limits
  • • DB 上限 10GB（Free 500MB）
  • • 底層 SQLite — single writer，寫入不能並行
  • • 單次 query 讀取上限 20MB，回傳上限 100MB
  • • 不適合 high-write / OLAP 場景
  • • Free: 100,000 rows read/day, 5M rows read/month
✅ 適合：CRUD app、blog、小型 SaaS
❌ 爆了就換：Turso, Neon, Supabase, PlanetScale

🟢 Production

⚙ Configuration

  • Runs on localhost:8787
  • Hot reload on file changes
  • Local D1/KV/R2 simulation (SQLite)
  • Data stored in .wrangler/state/
  • Secrets from .dev.vars file

💡 Key Points

  • No Cloudflare account needed
  • Uses workerd runtime (same as prod)
  • --remote flag to use real CF resources
  • --test-scheduled for Cron testing
  • Vitest with @cloudflare/vitest-pool-workers
$ npx wrangler dev --port 8787
# .dev.vars
API_KEY=dev-secret-key
DATABASE_URL=local-db

⚙ Configuration

  • Deploys as my-worker-staging
  • Route: staging.example.com/*
  • Separate KV/D1 namespace IDs
  • Real Cloudflare infrastructure
  • Per-env vars in wrangler.toml

💡 Key Points

  • Uses [env.staging] in config
  • Env-specific bindings are non-inheritable
  • Must redeclare KV/D1 bindings per env
  • Secrets set per env with --env
  • Test with real network conditions
$ npx wrangler deploy --env staging
$ npx wrangler secret put API_KEY --env staging

⚙ Configuration

  • Deploys as my-worker-production
  • Routes: example.com/*
  • Production KV/D1 namespace IDs
  • Custom domains supported
  • Full observability (tail, analytics)

💡 Key Points

  • Uses [env.production] in config
  • Multiple routes supported per env
  • Secrets managed separately from staging
  • Zero-downtime deployments
  • Instant global rollback via dashboard
$ npx wrangler deploy --env production
$ npx wrangler tail --env production

🔹 CLI Commands
wrangler dev --port 8787
Start local development server with hot reload
wrangler deploy --env production
Deploy Worker to specified environment
wrangler secret put API_KEY --env staging
Set encrypted secret for an environment
wrangler d1 execute my-db --file schema.sql
Run SQL against D1 database
wrangler kv key put --binding APP_KV key val
Write key-value pair to KV namespace
wrangler tail --env production
Stream real-time logs from deployed Worker
wrangler types
Generate TypeScript types from bindings config
wrangler d1 migrations apply my-db
Apply pending D1 schema migrations
🔐 Authentication & Team
Account Hierarchy
Account ← company / team (account_id in wrangler.toml)
├── Members ← people (each with own login)
│ ├── alice@co.com (Super Admin)
│ ├── bob@co.com (Admin)
│ └── charlie@co.com (Developer)
├── Workers, KV, D1, R2, Queues ...
└── Zones (domains)
ⓘ One person can own multiple Accounts (personal + work). All resources belong to the Account, not to individual members.
🖥️ Local Dev
$ wrangler login
# opens browser → OAuth
# token saved to
# ~/.wrangler/config/
Run once, auto-refreshes. Each team member uses their own login.
