🗃️ KV — Key-Value Store
  • • Persistent global key-value store
  • • Eventually consistent — 寫入後最多 60 秒全球生效
  • • 適合讀多寫少，不適合頻繁寫入
  • • 沒有 filter/query，只能 list({ prefix })
  • • 單個 value 上限 25MB
  • • Free: 1,000 writes/day, 100,000 reads/day
✅ 設定檔、feature flags、metadata
❌ session store、計數器、即時狀態
wrangler.toml
[[kv_namespaces]]
binding = "APP_KV"
id = "abc123"
Code
const val = await env.APP_KV.get("key");
await env.APP_KV.put("key", "val");
await env.APP_KV.put("k", "v", {
  expirationTtl: 3600
});
🗄️ D1 — SQLite Database
  • • 邊緣 SQLite 資料庫（serverless）
  • • DB 上限 10GB（Free 500MB）
  • • Single writer — 寫入不能並行
  • • 單次 query 讀取上限 20MB，回傳 100MB
  • • 支援 migrations（wrangler d1 migrations）
  • • Free: 100K rows read/day, 5M/month
✅ CRUD app、blog、小型 SaaS
❌ high-write / OLAP → 改用 Turso, Neon, Supabase
wrangler.toml
[[d1_databases]]
binding = "DB"
database_name = "my-db"
database_id = "uuid-here"
Code
const { results } = await env.DB
  .prepare("SELECT * FROM users WHERE id=?")
  .bind(userId)
  .all();
📦 R2 — Object Storage
  • • S3 相容的物件儲存，零 egress 費用
  • • 單檔上限 5TB，無總容量限制
  • • 支援 multipart upload
  • • 可開 public bucket（靜態網站 hosting）
  • • 支援 S3 API（現有工具可直接用）
  • • Free: 10GB storage, 10M reads/month
✅ 圖片、影片、檔案上傳、備份
❌ 不適合頻繁小檔讀寫（用 KV）
wrangler.toml
[[r2_buckets]]
binding = "BUCKET"
bucket_name = "my-bucket"
Code
await env.BUCKET.put("img/photo.jpg", file);
const obj = await env.BUCKET.get("img/photo.jpg");
const list = await env.BUCKET.list({
  prefix: "img/"
});
🧩 Durable Objects — Stateful Actors
  • • 有狀態的單例 actor，每個 ID 全球唯一
  • • Strong consistency（與 KV 不同）
  • • 內建 WebSocket 支援（聊天室、協作）
  • • 自帶 transactional storage
  • • 可設 alarm（定時喚醒）
  • • 最複雜的 binding，學習曲線高
✅ 即時協作、WebSocket、rate limiter、遊戲伺服器
❌ 簡單 CRUD（用 D1）
wrangler.toml
[durable_objects]
bindings = [
  { name = "ROOMS",
    class_name = "ChatRoom" }
]
Code
const id = env.ROOMS.idFromName("room-1");
const stub = env.ROOMS.get(id);
const res = await stub.fetch(request);
📬 Queues — Message Queue
  • • Pull/Push message queue
  • • 保證 at-least-once delivery
  • • 支援 batch consume（批次處理）
  • • 自動 retry + dead letter queue
  • • 最大 message size: 128KB
  • • Free: 1M messages/month
✅ 背景任務、email 發送、webhook 處理
❌ 即時通訊（用 Durable Objects + WebSocket）
wrangler.toml
[[queues.producers]]
binding = "MY_QUEUE"
queue = "my-queue"

[[queues.consumers]]
queue = "my-queue"
Code
// Producer
await env.MY_QUEUE.send({ type: "email", to: "a@b.c" });

// Consumer
async queue(batch, env) {
  for (const msg of batch.messages) {
    // process msg.body
    msg.ack();
  }
}
🔒 Secrets — Encrypted Environment Variables
  • • 加密存儲，不寫在 wrangler.toml（避免進 git）
  • • 程式中用 env.SECRET_NAME 取值（string）
  • • 不會出現在 Dashboard API 回傳中
  • • 每個 env（staging/prod）各自設定
  • • 本地開發用 .dev.vars 檔模擬
與 [vars] 取用方式相同，差別在加密存儲和不外洩
CLI
$ wrangler secret put API_KEY
$ wrangler secret put API_KEY --env prod
$ wrangler secret list
Code
const key = env.API_KEY;
await fetch(url, {
  headers: { Authorization: `Bearer ${env.API_KEY}` }
});
.dev.vars
API_KEY=dev-test-key
DB_URL=postgres://localhost/dev
📊 Analytics Engine — Event Logging
  • • Write-optimized 時序事件儲存
  • • 用 SQL API 查詢（類 ClickHouse）
  • • 每個 event 最多 20 個 blobs + 20 個 doubles
  • • 資料保留 90 天
  • • 適合自建 analytics（取代 GA）
  • • Free: 100K events/day
✅ Page views、API metrics、自訂事件
❌ 長期資料保存（90 天限制）
wrangler.toml
[[analytics_engine_datasets]]
binding = "ANALYTICS"
Code
env.ANALYTICS.writeDataPoint({
  blobs: ["/api/users", "GET"],
  doubles: [response.status, latencyMs],
  indexes: ["api"]
});
🤖 Workers AI — ML Inference
  • • 在邊緣執行 ML 模型（Cloudflare GPU）
  • • 支援 LLM、embedding、image、translation 等
  • • 模型列表：Llama, Mistral, Stable Diffusion 等
  • • 搭配 Vectorize 做 RAG
  • • 不需自己管 GPU / infra
  • • Free: 10K neurons/day
✅ 文字生成、翻譯、圖片辨識、embedding
❌ 大量推理需求（成本高）→ 用自架 GPU
wrangler.toml
[ai]
binding = "AI"
Code
const res = await env.AI.run(
  "@cf/meta/llama-3-8b-instruct",
  { messages: [
    { role: "user",
      content: "Hello" }
  ]}
);
⚡ Cache API — Built-in HTTP Cache
  • • 不需要 binding — caches.default 直接用
  • • 存 HTTP Response 物件
  • • 依 Cache-Control header 過期
  • • 隨時可能被清除（不保證持久）
  • • Per-colo（非全球一致）
與 KV 差異：Cache 是暫存 Response、會被清除；KV 是永久 key-value 儲存
Code（不需設定）
const cache = caches.default;
let res = await cache.match(req);
if (!res) {
  res = await fetch(origin);
  ctx.waitUntil(
    cache.put(req, res.clone())
  );
}
return res;

⚙ Configuration

  • Runs on localhost:8787
  • Hot reload on file changes
  • Local D1/KV/R2 simulation (SQLite)
  • Data stored in .wrangler/state/
  • Secrets from .dev.vars file

💡 Key Points

  • No Cloudflare account needed
  • Uses workerd runtime (same as prod)
  • --remote flag to use real CF resources
  • --test-scheduled for Cron testing
  • Vitest with @cloudflare/vitest-pool-workers
$ npx wrangler dev --port 8787
# .dev.vars
API_KEY=dev-secret-key
DATABASE_URL=local-db

⚙ Configuration

  • Deploys as creator-staging
  • Route: staging-creator.ing-studio.work/*
  • Config lives in wrangler.toml
  • Separate KV/D1 namespace IDs from prod
  • Developers deploy directly from local

💡 Key Points

  • Uses [env.staging] in config
  • Developer gets staging-only API token
  • No [env.production] in wrangler.toml
  • Devs can only deploy staging by design
  • Secrets set per env with --env
$ npx wrangler deploy --env staging
$ npx wrangler secret put API_KEY --env staging

⚙ Configuration

  • Deploys as creator-production
  • Route: creator.ing-studio.work/*
  • Config NOT in wrangler.toml
  • Route passed via CI/CD --route flag
  • Token stored in GitHub Secrets only

💡 Key Points

  • Deployed only via GitHub Actions
  • Merge to main → auto deploy production
  • Developers cannot deploy prod from local
  • Separate API token from staging
  • Zero-downtime, instant rollback via dashboard
# GitHub Actions only
$ npx wrangler deploy --env production --route "creator.ing-studio.work/*"

🔐 Authentication & Team
Account Hierarchy
Account ← company / team (account_id in wrangler.toml)
├── Members ← people (each with own login)
│ ├── alice@co.com (Super Admin)
│ ├── bob@co.com (Admin)
│ └── charlie@co.com (Developer)
├── Workers, KV, D1, R2, Queues ...
└── Zones (domains)
ⓘ One person can own multiple Accounts (personal + work). All resources belong to the Account, not to individual members.

🛡 Deploy Token Strategy
Problem: Cloudflare tokens can't restrict to a specific Worker
API Token permissions are scoped to Account/Zone level, not individual Worker names. A token with Workers Edit on your zone can deploy any Worker in that zone.
Solution: Config-level isolation
🟡 Developer (staging)
  • • Gets staging API token
  • • wrangler.toml only has [env.staging]
  • • [env.production] not in config
  • • --env production fails (no config)
  • • Deploys via npm run deploy:staging
🟢 CI/CD (production)
  • • Production token in GitHub Secrets
  • • Route passed via --route flag
  • • Triggered by merge to main
  • • No human has production token
  • • Separate secrets per environment
